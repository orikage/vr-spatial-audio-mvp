# 1. 背景と課題

## 1.1 VRにおける応援上映・ライブ体験の現状

応援上映やライブ会場のような体験をVR上で実現したいというニーズがある。求められる体験は以下の通り：

- VRヘッドセットを装着した状態で没入している
- 周りに大勢の人がいる感がある
- 近くの人の声はクリアに聞こえる
- 遠くの声は歓声・アンビエントとして場の熱量が伝わる
- 声出しもできる

## 1.2 既存プラットフォームの限界

**VRChatの現状：**

- 通常インスタンスの上限は80人（ハードキャップ）
- VRChat公式の実験（Instance Capacity Stress Tests）では2025年12月に256人同時接続を記録しているが、あくまで実験段階
- ボイスチャットは全接続者への音声配信が前提で、人数増加に対してコストが線形に増大する
- Udonからボイスの音声データにはアクセスできず、パラメータ調整止まり

**問題の本質：**

インスタンス上限数そのものではなく、**ボイスチャットのスケーリングアーキテクチャ**にある。「近くの人の声だけ届ける」設計にすればスケールするが、プラットフォームレベルで対応しないと実現できない。

## 1.3 プロジェクトの位置づけ

「技術的にこうすれば解決できる」という建設的な提案の根拠を作るため、自前でプロトタイプを実装する。目的は純粋な技術的好奇心。

# 2. プロジェクト方針

## 2.1 目的

「距離ベースの音声ルーティング＋遠方アンビエント合成」によって大規模同時接続時の音声体験問題が技術的に解決可能であることを、シンプルなプロトタイプで証明する。

## 2.2 技術スタック

| 項目 | 採用技術 | 理由 |
|------|---------|------|
| 音声処理 | Web Audio API | ブラウザネイティブ、追加ライブラリ不要 |
| ネットワーク | LiveKit（Phase 3以降） | OSSのSFU、距離減衰を自前実装できる |
| 3D空間 | Unity（Phase 4以降） | VRヘッドセット対応に必要 |
| MVP | ブラウザ単体 | サーバー不要、HTMLファイル1個で動作 |

## 2.3 Claude Codeでの進め方

各Phaseで動作確認してから次に進む。一気に全部作らせない。

- **Phase 1**：「Web Audio APIでPannerNodeとBiquadFilterNodeを使って距離減衰とローパスフィルタを実装して」
- **Phase 2**：「複数の音源ポインタを追加して、遠方のものをGainNodeでミックスするアンビエント合成を追加して」
- **Phase 3**：「LiveKitのNode.js SDKを使ってSFUサーバーに接続し、座標をData Channelで共有する実装をして」

# 3. MVP定義

## 3.1 成功条件

MVPの達成条件は以下の1点に絞る。

> **空間上を動く音源ポインタが、自分との距離に応じて「クリアな近傍ボイス」と「こもった遠方アンビエント」に切り替わること**

## 3.2 スコープ

| 項目 | 内容 |
|------|------|
| 対象 | Web Audio APIベースのローカル動作プロトタイプ |
| 人数規模 | 2〜5人（概念実証レベル） |
| 入力 | wavファイルを再生し続ける疑似音源ポインタ |
| 出力 | 距離に応じた音量減衰＋ローパスフィルタ処理 |
| VR対応 | 対象外（ブラウザ2D画面で確認） |
| ネットワーク | 対象外（ローカル動作のみ） |

VR空間・見た目・ネットワーク接続はすべてMVPスコープ外とする。音響処理の本質だけを最短で検証する。

## 3.3 技術的に証明できること

- 「群衆アンビエント合成はクライアントサイドで実装できる」
- 「SFUベースなら人数スケールは原理的に解決できる」

# 4. 音響モデル編

## 設計の出発点

応援上映・ライブ会場における「場の熱量」の知覚メカニズムを分解する。人間が「大勢いる」と感じるのは音量ではなく**音の複雑性と空間的拡散の組み合わせ**である。

- 近くの人の声が**個別に識別できる**
- 遠くの声が**方向感を持った拡散音**として聞こえる
- 全体が**ひとつの音響空間**として統合されている

この3条件を同時に満たすモデルを設計する。

---

## M0：座標系と単位の定義

### 音響空間の単位

すべての距離はメートル [m] を単位とする。

### Canvas座標との変換

Canvas上の1画素が何メートルに相当するかを`PIXELS_PER_METER`定数で管理する。

```javascript
const SCENE = {
  canvasWidth:      800,  // [px]
  canvasHeight:     600,  // [px]
  PIXELS_PER_METER: 20,   // 音響空間: 40m × 30m
};

// Canvas座標 → メートル変換（リスナーをCanvas中央に固定）
function toMeters(canvasPx, canvasPy) {
  return {
    x: (canvasPx - SCENE.canvasWidth  / 2) / SCENE.PIXELS_PER_METER,
    z: (canvasPy - SCENE.canvasHeight / 2) / SCENE.PIXELS_PER_METER,
  };
}
```

**スケール感の確認（ambientNoiseSPL=75dBの場合）：**

| 距離 | Canvas上での長さ |
|------|----------------|
| d_near = 3.2m | 64px |
| d_trans = 10m | 200px |
| 画面端 = 20m | 400px |

### リスナー位置

MVPではリスナーはCanvas中央 = (0, 0) m に固定する。向きはリスナードラッグUIで更新する（I4参照）。

### 高さ（Y軸）

MVPでは全音源をY=0（水平面）に固定する。PannerNodeのpositionYは常に0。

### Web Audio API座標系との対応

Web Audio APIはY軸上向きの右手系を使う。Canvas（Y軸下向き）との変換：

```
pannerNode.positionX.value =  meters.x
pannerNode.positionY.value =  0
pannerNode.positionZ.value = -meters.z   // Canvas Y軸を反転
```

## M1：音声の物理伝搬モデル

### 基本減衰（逆距離則）

```
gain_physical(d) = d_ref / d   (d > 0)
```

| 記号 | 意味 | 値 |
|------|------|----|
| d_ref | 基準距離（gain=1.0の参照点） | 1m |
| d | 音源〜リスナー間の距離 | [m] |

> **注意：** Web Audio APIのPannerNodeが持つ内部距離減衰は`rolloffFactor=0`に設定して**無効化**する。距離減衰はすべてGainNodeで自前計算する。PannerNodeは方向（panning）の処理のみに使う（詳細はI1参照）。

`rolloffFactor=0`にしない場合、PannerNodeの内部減衰と自前GainNodeが二重に適用されてしまう。

### SPLモデル（ゾーン境界導出に使用）

```
SPL(d) = SPL_ref - 20 × log10(d / d_ref)
```

| 記号 | 意味 | 値 |
|------|------|----|
| SPL_ref | 基準距離1mでの仮想音圧レベル（VRシステムのゲイン較正値） | 100dB（応援上映・ライブ体験向けに較正。物理的な会話音声 65dB ではなく、体験設計値として逆算した値） |

このSPLモデルはゾーン境界計算（M2）にのみ使う。実際の音量制御は`gain_physical`で行う。

### cutoff周波数の距離依存モデル（経験的近似）

**注意：** このモデルは物理的な空気吸収（ISO 9613-1）をそのまま使うのではなく、「遠距離ほどこもって聞こえる」知覚効果を再現するために現実よりも強調した経験的近似式を使う。

実際のISO 9613-1空気吸収値（20℃、50%RH）：

| 周波数 | α_air [dB/m] | 3dB吸収に必要な距離 |
|--------|-------------|-------------------|
| 1kHz | 0.0014 | 2143m |
| 4kHz | 0.0100 | 300m |
| 8kHz | 0.0260 | 115m |

実際の空気吸収は数百〜数千メートルスケールでしか聴感上の差を生まない。VR空間（〜数十m）でリアルな演出効果を得るため、**知覚的に自然なスケールに拡張した経験的モデル**を使う。

**経験的cutoff近似式：**

```
f_cutoff(d) = 30000 / d^(2/3)   [Hz]
```

| 距離 | f_cutoff |
|------|----------|
| 1m | 30,000Hz → **clampで20,000Hz適用**（Nyquist制限） |
| 3.2m (d_near) | 〜13,800Hz（微細な変化のみ） |
| 10m (d_trans) | 〜6,500Hz（明瞭な曇り） |
| 30m | 〜3,200Hz（はっきりこもる） |
| 100m | 〜1,400Hz（くぐもった声） |

**実装コード：**

```javascript
// 上限クランプ: BiquadFilterNodeはNyquist周波数（≈24000Hz）を超えると未定義動作になる
// 下限クランプ: 300Hz以下にはしない（過度に不自然になるため）
function fCutoff(d) {
  const raw = 30000 / Math.pow(Math.max(d, 0.1), 2/3);
  return Math.min(Math.max(raw, 300), 20000);
}
```

## M2：知覚的ゾーン境界の導出

### SN比による明瞭度判定

```
SNR(d) = SPL(d) - SPL_ambient
```

| SNR | 知覚状態 | ゾーン |
|-----|---------|--------|
| > 15dB | 個別に明瞭識別可能 | near |
| 5〜15dB | 部分的に識別可能 | transition |
| ≤ 5dB | 識別不能、拡散音として知覚 | ambient |

### ゾーン境界の計算式

```javascript
function deriveZones(ambientNoiseSPL) {
  // SPL_ref=100dB は物理的な会話音声（65dB）ではなく、
  // 応援上映・ライブ会場での歓声を再現するための VR ゲイン較正値。
  // この値を使うと各シナリオの期待ゾーン境界が得られる:
  //   ambientNoiseSPL=85dB → d_near=1.0m, d_trans=3.2m
  //   ambientNoiseSPL=75dB → d_near=3.2m, d_trans=10m
  //   ambientNoiseSPL=55dB → d_near=32m,  d_trans=101m
  const SPL_ref = 100;
  const d_ref   = 1;

  // SN比 15dB → nearゾーン終端
  const d_near  = d_ref * Math.pow(10, (SPL_ref - ambientNoiseSPL - 15) / 20);
  // SN比  5dB → transitionゾーン終端
  const d_trans = d_ref * Math.pow(10, (SPL_ref - ambientNoiseSPL -  5) / 20);

  // クランプ: 物理的に意味のない極値を防ぐ
  const d_near_clamped  = Math.max(d_near,  0.1);   // 最小0.1m
  const d_trans_clamped = Math.max(d_trans, 0.32);  // d_nearの最低3倍

  return {
    d_near:  d_near_clamped,
    d_trans: d_trans_clamped,
    f_near:  fCutoff(d_near_clamped),
    f_trans: fCutoff(d_trans_clamped),
  };
}
```

### ゾーン境界の比率は常に固定

```
d_trans / d_near = 10^((SPL_ref - ambient - 5 - (SPL_ref - ambient - 15)) / 20)
                = 10^(10/20)
                = sqrt(10)
                ≈ 3.16
```

**ambientNoiseSPLによらず、d_transは常にd_nearの√10倍。** これはSN比の閾値（5dBと15dBの差 = 10dB）から導かれる固定比率。

**シナリオ別ゾーン境界：**

SPL_ref=100dB（歓声・チャント参加者の VR ゲイン較正値）で計算した値:

| シナリオ | SPL_ambient | d_near | d_trans |
|---------|------------|--------|---------|
| ライブ会場 | 85dB | 1.0m | 3.2m |
| 応援上映 | 75dB | 3.2m | 10m |
| 静かな上映 | 55dB | 32m | 101m |

## M3：3ゾーンモデル

```
距離
0──────────d_near──────────d_trans──────────∞
    nearゾーン    transitionゾーン    ambientゾーン
```

### nearゾーン（0〜d_near）

クリアな個別ボイスとして聞こえる。

- 距離に応じて音量のみ変化（ローパスフィルタなし）
- HRTFによる立体音響で定位を明確にする
- プレゼンス強調（HighShelf +2dB @ 3kHz）で「前に出る」感じを出す
- 同時再生上限 N_max = `maxNearVoices`（デフォルト5）

**N_maxを超えた音源の扱い：**
nearゾーン内にいても、スコア（I5参照）が下位の音源はambient扱いにする。nearGain=0、transGain=0、indivAmbGain=gain_physical(d) を設定する（正規化はAmbGainが担当）。「声が聞こえなくなる」のではなく「遠くの背景に溶け込む」挙動にする。

**HighShelfの適用スコープ：**
HighShelfはnearパス（[HighShelf] → [HRTF Panner] → [NearGain]）のみに存在する。nearGainが0に近づくにつれて自然にフェードアウトするため、transitionゾーンでのHighShelf効果は自動的に消える。

### transitionゾーン（d_near〜d_trans）

nearとambientを知覚的に自然につなぐ。

- 音量・cutoff周波数・定位が連続的に変化
- smoothstep補間で境界でのカクつきを防ぐ
- ambientバスにも徐々に流して連続性を高める（M4参照）

### ambientゾーン（d_trans〜）

個別の声として認識されない。全員分を合成して群衆音として出力する。

- 仮想8音源に分散して方向感・包囲感を付与
- 密度補正で人数が増えるほど豊かに聞こえるようにする
- ConvolverNodeで残響を付与（任意）

## M4：遷移ゾーンの連続性モデル

### 正規化パラメータ

```javascript
// clamp必須: nearゾーン(d<d_near)でt<0、ambientゾーン(d>d_trans)でt>1になる
// clampなしの場合、nearGain/transGain が負や1超になり音量が発散する
const t   = Math.max(0, Math.min(1, (d - d_near) / (d_trans - d_near)));
const t_s = 3*t*t - 2*t*t*t;                      // smoothstep（端点で微分=0、clamp後なら 0≤t_s≤1）
```

### 各パラメータの補間

**基準gain値の定義：**

```javascript
const gain_near_at_boundary = 1 / d_near;   // nearゾーン端でのgain_physical
const gain_amb_base         = 1 / d_trans;  // ambientゾーン開始点でのgain_physical
```

`gain_amb_base`は物理距離減衰 `d_ref/d` を `d=d_trans` で評価したもの。この定義により、transitionゾーン終端とambientゾーン開始点のgain_physicalが一致し、ゾーン境界での音量の急変がない。

**音量（nearパスとtransパスのクロスフェード）：**

```javascript
// nearGain + transGain = gain_physical(d) を常に保つ
// → nearゾーン境界での+6dBジャンプが発生しない
nearGain.gain  = gain_physical(d) * (1 - t_s);  // t_s=0でフル、t_s=1でゼロ
transGain.gain = gain_physical(d) * t_s;         // t_s=0でゼロ、t_s=1でフル
```

**transitionからambientバスへの寄与量：**

```javascript
// 同じgain_physical(d)にt_sをかけて徐々に流し込む
indivAmbGain.gain = gain_physical(d) * t_s;
```

**Cutoff周波数（対数補間）：**

```javascript
// 人間の周波数知覚は対数的なので対数スケールで補間する
const cutoff = Math.exp(
  Math.log(f_near) * (1 - t_s) + Math.log(f_trans) * t_s
);
```

**Pannerのblend動作：**

transitionゾーンではnearパスとtransパスが同時に動作し、gainのクロスフェードで比率を制御する。

```javascript
// gain_physical(d) = 1/max(d, d_ref) で常に sum = gain_physical(d) を保つ
nearGain.gain  = gain_physical(d) * (1 - t_s);
transGain.gain = gain_physical(d) * t_s;
```

**なぜ sum = gain_physical(d) が重要か：**
- nearゾーンの出口（d=d_near, t_s=0）：nearGain=1/d_near、transGain=0 → 合計=1/d_near（near最終値と一致）
- ambientゾーンの入口（d=d_trans, t_s=1）：nearGain=0、transGain=1/d_trans（ambient開始値と一致）
- 途中（任意のd, t_s）：nearGain + transGain = gain_physical(d) = 1/d（距離減衰と常に一致）

この設計にすることで、ゾーン境界での音量ジャンプが完全に解消される。

**定位のぼやけ：**

```javascript
// transitionで定位が徐々にぼやける
const angle_noise = t_s * (Math.PI / 4) * xi(t);
// xi(t): 0.5〜3Hzの帯域制限ホワイトノイズ（低周波のゆらぎで「じわじわ」感を出す）
```

## M5：密度補正モデル

### N_ambの定義（旧方式・参考）

```
N_amb = ambientゾーンに分類された音源の数
      = (ambientゾーン内の音源数) + (nearゾーン内だがN_maxを超えて溢れた音源数)
```

MVPの2〜5音源ではN_amb=1〜4程度が現実的な範囲。将来的には100〜300を想定する。

> **設計アップグレード（N_eff を推奨）：** N_amb は整数カウントのため、transitionゾーンを通過するたびに不連続にステップする。これにより AmbGain が段階的に変化し、人数変動時に音量がカクつく。
> **より滑らかな実装** は `N_eff = Σ(t_s_i²)`（寄与率の二乗和）を使い `AmbGain = 1/√N_eff × 密度ブースト` とすること（MVP.md・ドメインロジック.md 参照）。
> N_eff はtransitionゾーン中間の音源を「0.25人分」などとして扱うため、人数増減が連続的に聞こえる。実装では N_amb の代わりに N_eff を使うこと。

### 設計上の重要な注意点：正規化はAmbGainマスターで行う

**N人のインコヒーレント音源を合算したバスのRMSレベル ≈ √N × 個別レベル。**

正規化 `1/√N` を各音源の `indivAmbGain` に掛けても、数学的には正しく機能する：

```
バスRMS = √N × (個別レベル × 1/√N) = 個別レベル  ← 正規化成立
```

ただし、N_ambが変化するたびに **全ambient音源の gain を再計算・再設定** する必要がある。
これは実装を複雑にし、複数ノードの更新タイミングのズレでクリックノイズが発生するリスクがある。

**そのため、正規化を AmbGain マスターノード1つに集約する設計を採用する。**

### アーキテクチャ：正規化は AmbGain マスターノードに集約する

```
[Source_1] → [indivAmbGain_1 = gain_physical(d1)] ─┐
[Source_2] → [indivAmbGain_2 = gain_physical(d2)] ─┤→ [AmbientBus] → [AmbGain] → [仮想8音源]
[Source_N] → [indivAmbGain_N = gain_physical(dN)] ─┘
```

**indivAmbGain（各音源、N_ambに依存しない）：**

```javascript
// 各音源は距離減衰のみ担当
indivAmbGain = 1 / Math.max(d, d_ref);
```

**AmbGain（マスター、N_eff変化時にこの1ノードだけ更新）：**

```javascript
// N_eff = Σ(t_s_i²) を使う（整数カウントのN_ambより滑らか。ドメインロジック.md 参照）
// エネルギー正規化 + 密度感ブースト
const gain_normalized = 1 / Math.sqrt(Math.max(N_eff, 1e-6)); // ゼロ除算ガード
// density_boost: N_eff≈16-17人で約+3dB、N_eff→∞で最大+6dBに飽和
// ※ tanh(1)≒0.76なのでN_ref=30の時点では+4.6dB（"半分"になるのはN≒16-17人）
const gain_density    = Math.pow(10, 6 * Math.tanh(N_eff / 30) / 20);
AmbGain = gain_normalized * gain_density;
```

**バスレベルの推移（この設計の場合）：**

バスレベル ≈ `√N_eff × avg(gain_physical)` （インコヒーレント合成）
AmbGain適用後 ≈ `√N_eff × avg(gain_physical) × (1/√N_eff) × gain_density`
              = `avg(gain_physical) × gain_density`

**結果：ambient全体レベルは人数によらずほぼ一定（avg距離減衰 × 密度ブーストのみ）。**

### スケール確認

**AmbGainノードの値（= 1/√N × density_boost）：**

| N_amb | 1/√N | density_boost | AmbGainノード値 | バス × AmbGain（最終出力レベル比） |
|-------|------|--------------|--------------|----------------------------------|
| 1 | 1.000 | 1.023 | 1.023 | √1 × 1.023 = **1.023**（基準） |
| 5 | 0.447 | 1.118 | 0.500 | √5 × 0.500 = **1.118**（+0.7dB） |
| 30 | 0.183 | 1.692 | 0.309 | √30 × 0.309 = **1.692**（+4.6dB） |
| 300 | 0.058 | 1.995 | 0.115 | √300 × 0.115 = **1.992**（+5.8dB） |

**読み方：** AmbGainノードの値は人数が増えると小さくなるが、それはバスの合算（√N倍）を打ち消すため。**最終出力レベルは基準から最大+6dBの範囲でしか変化しない**（density_boostの効果のみ）。near voiceが埋もれない。

### transitionゾーンのambient寄与（修正）

transitonゾーンからambientバスへの寄与も `indivAmbGain` 経由なので整合する：

```javascript
// transitionゾーン：距離減衰にt_sをかけて徐々に流し込む
indivAmbGain = (1 / Math.max(d, d_ref)) * t_s;
```

## M6：Ambientの空間表現モデル

### 役割の明確化

| 処理 | 担当する現象 | MVP優先度 |
|------|------------|---------|
| 仮想音源配置 | 空間軸の拡散（方向感・包囲感） | 必須 |
| ConvolverNode | 時間軸の拡散（残響・空間の広がり） | 任意 |

2つは独立した現象なので役割重複ではない。MVPでは仮想音源配置を優先する。

### 仮想音源の配置設計

**音源数の根拠：** 心理音響研究（Chowning 1971、Gerzon 1992のAmbisonic研究）により、4〜8チャンネルのサラウンドで十分な包囲感が得られることが示されている。8音源を採用する。

```javascript
const N_virtual = 8;
const r_v       = d_trans * 1.5;  // transitionゾーン外縁より少し遠い位置

// 仮想音源 i の位置
const theta_i = (2 * Math.PI / N_virtual) * i;  // i = 0..7
const pos_i = {
  x: r_v * Math.cos(theta_i),
  y: 0,
  z: r_v * Math.sin(theta_i),
};
```

`r_v = d_trans * 1.5`の根拠：ambientゾーン開始点（d_trans）より遠くに配置することで、仮想音源が「確実に遠い」と知覚される。

### 仮想音源あたりのgain

ambient bus（`gain_amb_final`が適用済み）を8チャンネルに分配するとき、エネルギーを保存するため√8で割る：

```javascript
const gain_per_virtual = 1 / Math.sqrt(N_virtual);  // = 1/√8 ≈ 0.354
```

**理由：** 8チャンネルのインコヒーレントな信号を合成すると振幅は√8倍になる。各チャンネルを1/√8にすることで合成後のエネルギーが入力と等しくなる。

### gainゆらぎ

```javascript
// VirtualGain_i = gain_per_virtual = 1/√8（固定）
// AmbGainで正規化・密度補正済みのバスを受け取るため、ここは分配係数のみ
// ゆらぎはVirtualGainに低周波変調として追加する
virtualGain_i(t) = gain_per_virtual * (1 + A * Math.sin(2*Math.PI * f_i * t + phi_i));

// パラメータ
const A    = 0.3;                      // 変調深度30%
const f_i  = random(0.3, 1.5);         // Hz（発話エンベロープの典型的変動周波数）
const phi_i = random(0, 2 * Math.PI);  // 初期位相をランダムに
```

> **実装注意（OscillatorNodeを使うこと）：**
> rAF（requestAnimationFrame）で毎フレーム `linearRampToValueAtTime` を呼ぶと、
> 60fps = 16ms ごとに新しいランプが以前のランプをキャンセルし、
> ゆらぎが「階段状」になる。sin波の形が失われてしまう。
>
> 正しい実装方法：`OscillatorNode` で sin 波を生成し、`GainNode.gain` に
> `AudioParam.connect(oscillatorNode)` で乗算変調する（オーディオグラフ内で解決）。
>
> ```javascript
> // 正しいゆらぎ実装
> const osc   = ctx.createOscillator();
> const depth = ctx.createGain();
> const vGain = ctx.createGain();
>
> osc.frequency.value = f_i;   // 0.3〜1.5Hz
> depth.gain.value    = A * gain_per_virtual;  // 変調深度
>
> vGain.gain.value = gain_per_virtual;  // DC成分（基準値）
> osc.connect(depth);
> depth.connect(vGain.gain);  // gain = gain_per_virtual + depth * sin(...)
> osc.start();
> ```

### ConvolverNode（任意実装）

パラメータを2つに絞る：

```javascript
const T60 = 1.5;   // 残響時間 [秒]（ライブ会場典型値：1.0〜2.5秒）
const DRR = -3;    // Direct-to-Reverberant Ratio [dB]（ライブ会場典型値：-3〜-6dB）
                   // 負値 = 残響 > 直接音（ライブ会場では残響優勢）
```

## M7：Gainキャリブレーション

### 目的

すべてのゾーンを通じて出力レベルが-6dBFS以内に収まることを保証する。

### 入力wav想定レベル

```
想定入力: -18dBFS (peak) の会話音声
```

### gain_physicalの実効値

`d_ref = 1m`, `gain_physical(d) = 1/d` のとき：

| 距離 | gain_physical | 減衰量 |
|------|--------------|--------|
| 1m (d_ref) | 1.000 | 0dB |
| 3.2m (d_near) | 0.313 | -10dB |
| 10m (d_trans) | 0.100 | -20dB |
| 40m (画面端) | 0.025 | -32dB |

### nearゾーンの最大出力

```
入力 -18dBFS + gain_physical(d_ref=1m) × HighShelf +2dB
= -18dBFS + 0dB + 2dB = -16dBFS (d=1m, 最大)
```

HighShelfは+2dBなのでクリッピングの心配はない。

### ambientゾーンの最大出力

```
N_amb=1 のとき最大: gain_physical(d_trans=10m) = 0.1 = -20dB
入力 -18dBFS × 0.1 × gain_density(1) = -18dBFS - 20dB + 0.2dB ≈ -38dBFS
```

ambientゾーンは常に大きなマージンがあり、clippingの心配はない。

### masterGain

MVPでは`masterGain = 1.0`（調整不要）。ただし今後複数音源が同時にnearゾーンに入る場合は`1/sqrt(maxNearVoices)`程度に下げることを検討する。

## I1：ノードグラフ設計

```
[Source]
  ├→ [HighShelf +2dB @ 3kHz] → [HRTF Panner (rolloff=0)]  → [NearGain]  ─→ [Master] → [Out]
  ├→ [LowPass(可変cutoff)]   → [Equal Panner (rolloff=0)] → [TransGain] ─→ [Master]
  └→ [IndivAmbGain = gain_physical(d) × (0 or t_s or 1)]
         ↓
    [AmbientBus] ← 全ambient音源がここに合算される
         ↓
    [AmbGain = gain_normalized(N_amb) × gain_density(N_amb)]
         ↓
    [VirtualGain_0..7 = 1/√8] → [VirtualPanner_0..7 (rolloff=0)]
                                       ↓
                              [ConvOrBypass] → [Master]
```

**ノードの役割分担：**

| ノード | 設定値 | 更新タイミング |
|--------|-------|--------------|
| IndivAmbGain | `gain_physical(d) × zone_factor` | 音源が動くたびに更新 |
| AmbGain | `gain_normalized(N_amb) × gain_density(N_amb)` | **N_ambが変化したときだけ更新** |
| VirtualGain_0..7 | `1/√8`（固定） | 初期化時のみ |


### ゾーン別gainの制御

ゾーン切り替えはgainで制御し、ノードグラフ接続は変更しない（`disconnect()`/`connect()`は使わない）。グリッチを防ぐため`linearRampToValueAtTime`を使う。

> **実装注意（rAF + linearRampToValueAtTimeの干渉）：**
> `updateSourceGains` をrAF（16ms周期）から呼ぶと、RAMP=50msのランプが毎フレーム
> 上書きキャンセルされる。音源が静止している場合は同じ値へのランプが繰り返されるだけなので
> 実害は少ないが、急激な移動時には「目標値に到達する前に次の目標値へランプが始まる」現象が
> 起きる可能性がある。実聴で問題がなければそのままでよい。気になる場合は
> `setTargetAtTime`（指数的スムージング）への変更を検討する。

**AmbGainの更新はN_ambが変化したタイミングのみ行う（音源単位ではなくシステム全体で1回）。**

```javascript
const RAMP = 0.05; // 50ms

// ─── 音源単位の更新 ───────────────────────────────────────
function updateSourceGains(source, ctx) {
  const { d, zone, t_s } = source;
  const t = ctx.currentTime;
  const g_phys = 1 / Math.max(d, PHYSICS.dRef);

  if (zone === 'near') {
    source.nearGain.gain.linearRampToValueAtTime(g_phys, t + RAMP);
    source.transGain.gain.linearRampToValueAtTime(0,     t + RAMP);
    source.indivAmbGain.gain.linearRampToValueAtTime(0,  t + RAMP);

  } else if (zone === 'transition') {
    // nearGain + transGain = g_phys を保つクロスフェード
    // （両方フルゲインにすると境界で+6dBジャンプするバグを防ぐ）
    source.nearGain.gain.linearRampToValueAtTime(g_phys * (1 - t_s), t + RAMP);
    source.transGain.gain.linearRampToValueAtTime(g_phys * t_s,      t + RAMP);
    source.indivAmbGain.gain.linearRampToValueAtTime(g_phys * t_s,   t + RAMP);

  } else { // ambient（またはnear溢れ）
    source.nearGain.gain.linearRampToValueAtTime(0,      t + RAMP);
    source.transGain.gain.linearRampToValueAtTime(0,     t + RAMP);
    source.indivAmbGain.gain.linearRampToValueAtTime(g_phys, t + RAMP);
  }
}

// ─── システム全体のAmbGain更新（N_eff変化時のみ呼ぶ） ─────
// N_eff = Σ(t_s_i²) を渡すこと（整数カウントのN_ambは使わない）
function updateAmbGain(ambGainNode, N_eff, ctx) {
  const gain_normalized = 1 / Math.sqrt(Math.max(N_eff, 1e-6)); // ゼロ除算ガード
  const gain_density    = Math.pow(10, 6 * Math.tanh(N_eff / 30) / 20);
  ambGainNode.gain.linearRampToValueAtTime(gain_normalized * gain_density, ctx.currentTime + RAMP);
}
```

## I5：near優先度スタック

```javascript
class NearPriorityStack {
  constructor(maxVoices) {
    this.maxVoices    = maxVoices;
    this.lastActivity = new Map(); // sourceId → timestamp
  }

  // MVPではwavループなので全音源が常にactive（score = 1.0 / distance）
  // 将来的にVAD（Voice Activity Detection）に置き換える
  updateActivity(sourceId) {
    this.lastActivity.set(sourceId, performance.now());
  }

  score(sourceId, distance) {
    const age      = performance.now() - (this.lastActivity.get(sourceId) ?? 0);
    const activity = age < 500 ? 1.0 : 0.3; // 500ms以内の発話を優先
    return activity / Math.max(distance, 0.1);
  }

  // nearゾーン内の音源を分類する
  // 戻り値: { activeNear: [...], overflow: [...] }
  classify(nearSources) {
    const sorted = nearSources
      .map(s  => ({ ...s, score: this.score(s.id, s.distance) }))
      .sort((a, b) => b.score - a.score);

    return {
      activeNear: sorted.slice(0, this.maxVoices),
      overflow:   sorted.slice(this.maxVoices), // ambient扱いにする
    };
  }
}
```

## I6：合成IRの生成（ConvolverNode用）

```javascript
function createSyntheticIR(ctx, T60, drr) {
  const sampleRate = ctx.sampleRate;
  const length     = Math.floor(sampleRate * T60 * 1.5);
  const ir         = ctx.createBuffer(2, length, sampleRate);

  for (let ch = 0; ch < 2; ch++) {
    const data = ir.getChannelData(ch);

    // Schroederモデル：ホワイトノイズ × 指数減衰
    for (let i = 0; i < length; i++) {
      const t     = i / sampleRate;
      const decay = Math.exp(-6.91 * t / T60); // -60dBまでの減衰（6.91 ≈ ln(10^3)）
      data[i]     = (Math.random() * 2 - 1) * decay;
    }

    // DRR適用：先頭5msを直接音として扱い、DRRに応じてスケール
    // DRR = Direct/Reverberant比 [dB]
    // DRR=-3dB → 残響 > 直接音（ライブ会場的）→ drrLinear < 1 → 直接音を減衰
    // DRR=+3dB → 直接音 > 残響（近距離的）→ drrLinear > 1 → 直接音を強調
    const directSamples = Math.floor(0.005 * sampleRate);
    const drrLinear     = Math.pow(10, drr / 20);
    for (let i = 0; i < directSamples; i++) {
      data[i] *= drrLinear;
    }
  }

  return ir;
}
```

## I8：実装上の注意点

### AudioContextのサンプリングレート

デフォルトは44100Hzまたは48000Hz。HRTFの精度に影響するので48000を明示的に指定する。

### PannerNodeの距離減衰を必ず無効化する

```javascript
// ★ 必須: rolloffFactor=0にしないと距離減衰が二重にかかる
panner.rolloffFactor = 0;
```

自前のGainNodeで距離減衰を計算しているため、PannerNodeの内部距離減衰は不要。`rolloffFactor`デフォルト値は1なので、明示的に0にする必要がある。

### PannerNodeの座標系

Web Audio APIはY軸が上方向の右手座標系。Canvas（Y軸下向き）とは変換が必要。

```javascript
pannerNode.positionX.value =  meters.x;
pannerNode.positionY.value =  0;
pannerNode.positionZ.value = -meters.z; // Y軸を反転
```

### ノードグラフの動的変更

音源がゾーン切り替えするとき`disconnect()`/`connect()`は使わない。常時全ノードを接続しておき、gainで0にする方式を採る。

### 複数クライアントのテスト

一人では限界があるのでbotかTailscale越しに別端末を使う。

- マウスまたはアニメーションで音源ポインタを移動
- 距離計算 → PannerNodeの位置を更新
- ゾーン判定 → NearGain/TransGain/IndivAmbGainを更新

**確認ポイント：**
- 近くと遠くで明確な音の差があるか
- 移動に追従してリアルタイムに変化するか
- ローパス適用でこもり感が出ているか

## Phase 2 詳細

**実装内容：**
- 音源ポインタを5〜10個に増やし、それぞれが独立して移動
- 遠方ポインタの音声をambientバスでミックスして仮想8音源に分配
- UIに距離ごとの音源状態を表示（近傍/遠方の色分けなど）
- N_amb変化に応じてgain_normalizedとgain_densityを再計算

**確認ポイント：**
- 複数音源が混在しても近傍ボイスが埋もれないか
- 遠方アンビエントが群衆感として聞こえるか
- 音源数を増やしてもCPU負荷が許容範囲内か

## Phase 3 詳細

**実装内容：**
- Proxmox上のLXCかDockerでLiveKitサーバーを立てる
- Tailscale経由で外部アクセスできるようにする
- Node.js SDKでSFUに接続
- 座標をData Channelで共有
- クライアント側で座標を受け取り音量制御

**注意：** LiveKit自体には距離減衰なし。クライアント側で座標共有→音量制御を自前実装する。

# 7. 未解決の設計問題

レビュー・実測で検証が必要な項目をまとめる。

## 音響モデル編

| # | 問題 | 判断が必要な理由 | 状態 |
|---|------|----------------|------|
| 1 | SPL_refをシナリオ別に可変にすべきか | 叫び声は会話より15〜20dB高く、ゾーン境界が大きく変わる | 未解決 |
| 2 | SN比モデルの精度は十分か | STIの簡略版。精度差が体験に影響するか不明 | 未解決 |
| 3 | 室内補正（r_c）をMVPに入れるべきか | 大会場では逆距離則だけでは遠距離が静かすぎる可能性 | 未解決 |
| 4 | f_cutoffの経験的モデルの妥当性 | 30000/d^(2/3) は知覚的に良い効果を出すが、物理的根拠なし。実聴で調整が必要 | 要実測 |

## 実装設計編

| # | 問題 | 判断が必要な理由 | 状態 |
|---|------|----------------|------|
| 5 | HRTF→equalpowerの切り替えグリッチ | 50msランプで知覚的にマスクできるか実測が必要 | 要実測 |
| 6 | 仮想音源ゆらぎ周波数（0.3〜1.5Hz）の妥当性 | 発話エンベロープ参考値だが実測で検証が必要 | 要実測 |
| 7 | nearゾーン内複数音源の合成レベル | N_max=5が同時に最大音量になったとき総出力が過大にならないか | 要実測 |
| 8 | ConvolverNodeの負荷（Quest） | Oculus Browser上での実負荷未検証 | 要実測 |
| 9 | wavループ音源の位相コヒーレンス | 同じwavファイルをN本同時に再生すると位相が揃ってインコヒーレント仮定（√N則）が崩れる。対策：音源ごとに異なるwavを使う、または再生開始にランダムオフセットを入れる | Phase 2で検証 |
| 10 | Web Audio API の汎用HRTFによる頭内定位 | ChromeのHRTFは個人差非考慮のダミーヘッドモデル。「あそこから聞こえる」感が出ずに「頭の中で鳴っている」と感じる可能性がある。MVP段階では許容するが、Phase 4（VR対応）では個人化HRTFを検討する | MVP制約として記録 |

## VRChat・プラットフォーム編

| # | 問題 | 判断が必要な理由 | 状態 |
|---|------|----------------|------|
| 11 | VRChat Udon非互換 | BiquadFilterNode（ローパスフィルタ）・カスタムGainNodeルーティングはUdonからは実装不可。ブラウザプロトタイプで技術的可能性を「証明する」ことはできるが、VRChatへの移植にはプラットフォームレベルの対応が必要（プラットフォームへの技術的提案の根拠として使う） | 既知の制約 |
| 12 | Phase 3 ambientミキシングのサーバー/クライアント分担 | LiveKit SFUはトラックのルーティングのみ。ambient合成（AmbGain、仮想8音源）はクライアント側で実装する設計になっているが、300人規模のとき全音源のトラックをクライアントが受信・処理するのか、サーバー側でミキシングするのかが未定。スケール要件によって実装が大きく変わる | Phase 3設計時に決定 |

## 成功基準編

|---|------|----------------|------|

# 8. パラメータ一覧

| パラメータ | 分類 | 値 | 根拠 |
|-----------|------|-----|------|
| sampleRate | 固定 | 48000Hz | HRTF精度 |
| SPL_ref | 固定 | 100dB | VRゲイン較正値（物理的な会話音声 65dB ではなく、応援上映・ライブ体験の期待ゾーン境界から逆算した値。MVP.md 参照） |
| dRef | 固定 | 1m | gain_physical=1.0の参照距離 |
| rolloffFactor | 固定 | 0 | PannerNode距離減衰を無効化（自前計算） |
| f_cutoff定数 | 固定 | 30000 | 経験的近似（知覚的遠距離効果優先） |
| f_cutoff指数 | 固定 | 2/3 | d^(-2/3)則 |
| f_cutoff下限 | 固定 | 300Hz | 過度の不自然さを防ぐクランプ |
| highShelfFreq | 固定 | 3000Hz | プレゼンス帯域 |
| highShelfGain | 固定 | +2dB | 前に出る感じ |
| panSpreadMax | 固定 | π/4 (45°) | 定位ぼやけの上限 |
| virtualSourceCount | 固定 | 8 | Ambisonics研究 |
| virtualRadiusMult | 固定 | 1.5 | d_trans × 1.5（確実に遠い位置に配置） |
| gain_per_virtual | 固定 | 1/√8 ≈ 0.354 | 8チャンネル分配時のエネルギー保存 |
| modulationDepth | 固定 | 0.3 (30%) | 群衆の自然なゆらぎ |
| modulationFreqRange | 固定 | 0.3〜1.5Hz | 発話エンベロープ |
| densityBoostMax | 固定 | 6dB | 知覚的密度感の上限 |
| densityBoostRef | 固定 | 30人 | 密度感が飽和する人数 |
| PIXELS_PER_METER | 固定 | 20px/m | 40m×30mの音響空間をCanvas上に表現 |
| ambientNoiseSPL | ユーザー設定 | 75dB（デフォルト） | 応援上映の典型値 |
| maxNearVoices | ユーザー設定 | 5 | 混濁しない上限の経験則 |
| d_near | 導出 | ambientNoiseSPLから計算（最小0.1m） | SN比モデル |
| d_trans | 導出 | d_near × √10（最小0.32m） | SN比モデル、d_nearとの比率固定 |
| f_near | 導出 | 30000 / d_near^(2/3) | 経験的cutoffモデル |
| f_trans | 導出 | 30000 / d_trans^(2/3) | 経験的cutoffモデル |
| gain_near_at_boundary | 導出 | 1 / d_near | 逆距離則 |
| gain_amb_base | 導出 | 1 / d_trans | transitionゾーン終端のgain_physical（参照値。AmbGainとは別） |
| AmbGain | 導出 | gain_normalized(N_eff) × gain_density(N_eff) | N_eff変化時にのみ更新するマスターgain（N_eff = Σ(t_s_i²)） |
